Redis是开源的,内存中的数据结构存储系统,可以用作数据库,缓存,消息中间件.

Redis基本数据类型

参考

String 字符串

String字符串是字节序列,是二进制安全的,这意味着他们有一个已知的长度没有任何特殊字符终止,所以可以存储任何东西,512M为上限,主要是操作键值对;String的数据结构是简单的key-Value模型,Value可以是字符串.也可以是数字.

Redis使用的是自己构建的一种简单动态字符串,不光可以保存文本数据还可以保存二进制数据,并且获取字符串长度复杂度为,除此之外,Redis的SDS API是安全的,不会造成缓冲区溢出.

应用场景

  • 存储常规数据的场景
    • 缓存Session,Token,图片地址,序列化后的对象(相比较于Hash存储更节省内存)
  • 存储计数的场景
    • 用户单位时间的请求数(简单限流可以用到).页面单位时间的访问数
  • 分布式锁
    • 利用SETNX key value命令实现一个最简易的分布式锁(存在缺陷,不建议)

List列表

Redis的list是链表数据结构的实现,由Redis自己实现的双向链表结构;支持反向查找和遍历,方便操作,但是也带来了部分额外的内存开销.

应用场景

  • 信息流展示
    • 最新文章,最新动态
    • 即利用lrange进行分页展示
  • 消息队列
    • 功能简单且存在很多缺陷
    • Redis5.0之后新增数据结构Stream,也适合作为消息队列,但是功能依旧简陋

Hash散列表

Redis的哈希是键值对集合;Redis的哈希值是字符串字段和字符串值之间的映射,因此常用来表示对象,还有经常变动的信息.

Hash更适合对象的存储,String更适合字符串存储.

应用场景

  • 对象数据存储场景
    • 如: 存储用户信息,商品信息,文章信息,购物车信息等

Set集合

Redis中的Set类型是一种无序集合,集合中元素没有先后顺序且是唯一的,类似于Java中的HashSet

当需要存储一个集合列表数据,又不希望出现重复数据时,Set是一个很好的选择,并且Set提供了可以判断某元素是否在一个Set集合内的重要接口

Set可以实现交集,并集,差集操作,比如:将一个用户的关注人存在一个集合中,将所有粉丝存在另一个集合,通过求交集则可以得到共同关注,共同粉丝等功能.

应用场景

  • 需要存放的数据不能重复的场景
    • 网站UV统计(数据量巨大的场景HyperLogLog更合适),文章点赞,动态点赞
  • 需要获取多个数据源交集,并集和差集的场景
    • 如: 共同好友,共同粉丝,好友推荐(差集),音乐推荐(差集),订阅号推荐(差集+交集)
  • 需要随机获取数据源中的元素的场景
    • 举例:抽奖系统,随机点名

Sorted Set(有序集合)

类似于Set;和Set相比,增加了权重参数score,使集合中的元素能按照score及进行有序排列,还可以通过score的范围来获取元素的列表

应用场景

  • 需要随机获取数据源中的元素根据某个权重进行排序的场景
    • 各种排行榜
  • 需要存储的数据有优先级或重要程度的场景
    • 如: 优先级任务队列

Bitmap位图

Bitmap 不是Redis中的实际数据类型,而是在String类型上定义的一组面向位的操作,将其视为位向量

由于字符串是二进制安全的块,且最大长度为512MB,它们适合用于设置最多个不同的位

Bitmap存储的是连续的二进制数字(0或1),通过Bitmap,只需要一个bit位来表示某个元素对应的值或者状态,key为对应元素本身

因为8个 bit 才组成一个 byte,所以 Bitmap本身会极大的节省存储空间

可以将Bitmap看作是一个存储二进制数字(0或1)的数组,则数组中的每个元素下标叫做offset(偏移量)

应用场景

  • 需要保存状态信息(0/1 即可表示)的场景
    • 用户签到情况,活跃用户情况,用户行为统计(比如是否点赞过某个视频)

HyperLogLog(基数统计)

是一种有名的基数技术概率算法,基于LogLog Counting(LLC)优化改进得来,并不是Redis特有;Redis是实现了该算法并提供了一些开箱即用的API.

Redis提供的HyperLogLog占用空间非常小,只需要 12k 的空间就能存储接近个不同的元素;并且,Redis对HyperLogLog的存储结构进行了优化,采用两种方式计数:

  • 稀疏矩阵: 计数较少时,占用空间小
  • 稠密矩阵: 计数达到某个阈值时,占用 12k 的空间

基数计数概率算法为了节省内存,并不会直接存储元数据,而是通过一定的概率统计方法预估基数值(集合中包含元素的个数)

因此HyperLogLog的计数结果并不是一个精确值,存在一定误差(标准误差为 0.81%)

应用场景

  • 数据量巨大(百万,千万级别以上)的计数场景
    • 如: 热门网站每日/每周/每月访问ip数统计,热门帖子UV统计

Geospatial(地理位置)

Geospatial index (地理空间索引,简称CEO) 主要用于存储地理位置信息,基于Sorted Set实现

通过CEO我们可以轻松实现两个位置距离的计算,获取指定位置附近元素等功能.

通过Redis可视化工具查看personLocation,可以发现,Geospatial底层就是Sorted Set

GEO 中存储的地理位置信息的经纬度数据通过GeoHash算法转换成了一个整数,该整数作为Sorted Set的score(权重参数)使用.

应用场景

  • 需要管理使用地理空间数据的场景
    • 如: 附近的人

Redis过期策略

有三种:

  1. 定时删除(设置过期时间)
  2. 惰性删除
  3. 定期删除

定时删除: 即设置某些key的过期时间,同时创建一个定时器,当到达过期时间时,删除key

  • 优点: 在过期时间时,能快速释放内存空间
  • 缺点:
    • CPU资源紧张时,若到达过期时间,删除这些key会导致出现
    • 创建定时器需要消耗一定时间,过多的定时器会影响性能

Redis过期策略主要是: 定期删除+惰性删除

  • 惰性删除: 指的是当Redis中存在一些过期的key时,不会主动去删除,而是当这些过期的key被访问时,Redis才会检查这些key并进行删除
    • 优点: 节省CPU资源,因为只有在必要的时候才会删除过期的key
    • 缺点: 已经过期的key可能会一直保留在内存中,占用内存资源
  • 定期删除: 指的是在指定时间间隔内,随机检查设置了过期时间的key,若存在已经过期的key,则执行删除,且过程由一个后台线程负责;
    • 优点: 定期清除过期的key,节省内存空间,确保内存的及时释放
    • 缺点: 执行频率过高,可能会占用大量CPU资源
  • 两者结合: 可以更有效的实现内存管理
    • 好处:
      • 惰性删除,可以保证被访问的key,如果过期则会被删除
      • 定期删除,可以保证,不被经常访问的key,可以被及时删除

Redis淘汰策略

Redis是一个基于内存的数据库,当存储的key达到内存最大时,再有新的key需要存储,则会出现无法存储的情况,此时需要有一种淘汰策略,删除一些冗余在内存中的key.

在Redis中一共有以下6种淘汰策略:

  • noeviction(默认缓存策略): 当存储的key达到内存最大时,继续Redis会返回错误,停止存储新的key.
  • allkeys-lru: 对于所有的key,删除**最近(指的是离过期时间最近)**访问次数最少的key(lru是计算每个key被访问次数的算法)
  • allkeys-lfu: (Redis4.0后新增的内存淘汰策略),对于所有的key,删除访问次数最少的key
  • allkeys-random: 对于所有的key,执行随机删除,即随机删除一个内存中的key
  • violate-lru: 对于设置了过期时间的key,删除**最近(指的是离过期时间最近)**访问次数最少的
  • violate-lfu: (Redis4.0后新增的内存淘汰策略),对于设置了过期时间的key,删除访问次数最少的key
  • violate-random: 对于设置了过期时间的key,执行随机删除
  • violate-ttl: 对于设置了过期时间的key,删除最接近过期时间(最快过期的)的key

对于violate-lru、violate-random、violate-ttl三种策略,当找不到可以删除的key时,会变成默认策略(noeviction).

缓存处理流程

即前台发送查询数据请求,后端先从缓存中查找数据,若缓存中没有数据,则继续到数据库中进行查找,查到数据后对缓存进行更新,并返回查到的数据;若没有查到数据,则返回空.

Redis缓存穿透

概念

即当用户发起请求查询一个数据时,若缓存中不存在该数据,即缓存没有命中,则会继续向数据库中查询,若数据库也没有,则本次查询失败.

当用户很多(秒杀场景)时,发送查询请求,如果缓存都没有命中,则这些请求都会去查询数据库,对数据库造成了很大的压力,即缓存穿透.

即缓存穿透是: 用户或者是攻击者不断发起请求查询缓存和数据库中都不存在的数据,最终导致数据库压力过大,发生崩溃.

解决方案:

布隆过滤器

布隆过滤器是一种数据结构,对所有有可能查询的参数以hash的方式存储,在控制层先进行校验,不符合则丢弃,从而避免了对底层存储系统的查询压力.

缓存空对象

缓存空对象是,在缓存未命中时,查询数据库,若返回空值,则将空值存入缓存中,
